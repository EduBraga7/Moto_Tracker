# Regras do Projeto: Moto Tracker (SaaS Edition)

## 1. Idioma e Ortografia (Estritamente pt-BR para UI)
- Todo texto visível para o usuário (HTML, alertas, mensagens de erro, logs de interface) DEVE ser escrito em Português do Brasil (pt-BR) impecável, com uso correto de acentos e pontuação.
- Sempre utilize `<meta charset="UTF-8">` no `<head>` dos arquivos HTML.
- **REGRA DE CÓDIGO:** Variáveis, nomes de funções, tabelas e colunas de banco de dados NUNCA devem ter acentos ou cedilha (ex: use `km_atual` e não `quilômetro`, use `manutencao` e não `manutenção`).

## 2. Lógica de Negócio e Nomenclatura
- **Abastecimento:** O cálculo de consumo DEVE sempre ser feito utilizando o "Odômetro Total" (KM do painel) subtraindo o último abastecimento. NUNCA utilize a lógica de "Trip zerado".
- **Peças Dinâmicas:** NÃO "chumbe" peças no código (como "Óleo" ou "Relação"). O sistema deve estar preparado para o usuário cadastrar qualquer peça e seu respectivo intervalo de troca.
- **Multi-veículos:** O sistema nasce focado em motos (especialmente a experiência de uma Bros 150), mas a arquitetura de dados deve tratar como um "veículo" genérico para futura expansão (carros/caminhões).
- Use "KM" (maiúsculo) para quilometragem, "Custo/KM" e "km/L".

## 3. Arquitetura, Clean Code e Segurança
- **Stack Atual:** Python (Flask), Firebase (Firestore/Auth) e Vercel Serverless.
- **Clean Code:** Escreva um código autoexplicativo. Nomes de variáveis devem explicar *o que* o código faz. Só utilize comentários para explicar o *porquê* de uma decisão de negócio (ex: uma margem de segurança matemática).
- **Segurança (LGPD):** NUNCA inclua dados sensíveis (como placa do veículo) nos exemplos de código. Utilize o `user_id` para garantir o isolamento total dos dados (Multi-tenant) no banco.
- **Credenciais:** NUNCA deixe chaves de API, senhas ou tokens hardcoded. Utilize sempre o arquivo `.env` (`os.environ.get()`).
- NUNCA use `except: pass`. Sempre faça o tratamento e o log do erro adequadamente para evitar retornos 500 silenciosos.
- As rotas do Flask devem ser enxutas, chamando regras de negócio isoladas na camada de `services`.

## 4. UX & Frontend (Mobile First)
- **Estilo de Design:** "Moto-Noir Digital" (Dark mode, glassmorphism, glowing accents em ciano e laranja).
- **Mobile First:** Toda interface deve ser desenhada primeiramente para telas de celular, com botões e formulários grandes (touch-friendly), pensando no usuário que está no posto de gasolina ou na oficina.
- Entradas de dados devem ser "conversacionais" e divididas de forma lógica para evitar erros de digitação (ex: o formulário de abastecimento deve ter inputs claros e separados).
